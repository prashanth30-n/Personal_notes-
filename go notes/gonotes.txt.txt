//channels
go philosophy tells that communicate by passing data through channels like lets say we have a channel which is unbuffered no size example for this program is below
func main() {
    ch := make(chan int)

    go func() {
        ch <- 10
    }()

    x := <-ch
    fmt.Println(x)
}
in this code the main go routine blocks until it receives a value from other go routine which sends the data//the blocking happens until some communication happens between them like they can communicate with data through channels or like they can send the signal close the to main and they internally works like sending the data through channels but no need of data just a close is enough this pattern you can see at server and graceful shutdown 
this is how the unbuffered channels works
///buffered channels there is a size for channels in this context
func main() {
    ch := make(chan int, 2)

    go func() {
        ch <- 1
        ch <- 2
    }()

    x := <-ch
    y := <-ch

    fmt.Println(x, y)
}
in this the blocking happens for main go routine unitl there is some value from the receiver side Main receives one by one and sender blocks after the buffer size is full

Buffered channel capacity affects SEND blocking, not RECEIVE blocking

Receive cares only:

empty → wait

not empty → continue
